# ADR-007: Keep UI Evolution Separate From v1.0 Stability

## Purpose
Record the UI evolution and release-stability decision.

## Background
The QA brief explicitly requires design exploration not to risk v1.0 stability.

## Current State
SNTO's current UI is demonstration-rich and too broad for daily operations, but v1.0 is treated as deployed.

## Evidence
Product, director, and UX reviews recommended decision-first UI evolution while the repository audit warned about branch and merge risk.

## Problem
Aggressive redesign during stabilization could destabilize the current release and blur strategic priorities.

## Context
The current product must preserve v1.0 continuity while future versions move toward an operational decision console.

## Options Considered
- Redesign the current dashboard immediately.
- Freeze all UI work.
- Separate stabilization from UI evolution and stage UI changes by release.

## Decision
v1.1 should stabilize and clarify. v1.2 may introduce controlled decision views. v2.0 should own modular product architecture and major UI evolution.

## Consequences
Positive: protects v1.0 while enabling product maturity. Negative: visible UX transformation takes longer.

## Status
Accepted from 2026 QA and consensus alignment.
