# ADR-012: Deploying the /api/v2 HTTP Surface

## Purpose
Scope how (and when) the FastAPI `/api/v2` surface built in Fase 5 gets
deployed as a reachable HTTP service, closing the gap found during the
2026-07-18 production cutover: the API exists as code + tests only.

## Background
Fase 5 (ADR-011, PRs #62–#70) delivered a complete read+write `/api/v2`
surface over the persistence layer: managed assets, observations, alerts,
recommendations, interventions with validated lifecycle transitions, field
verifications, an audit log, and minimal API-key write-auth. The production
Postgres cutover (§4bis of the Fase 5 plan) was executed on 2026-07-18 and
verified live.

During that cutover the owner found that **`src/api/main.py` is not deployed
anywhere**. The only deployed artifact is the Streamlit Container App
(`snto-observatory`), whose Dockerfile `CMD` runs `streamlit run app.py` on
port 8501. The "Acciones Urgentes" tab consumes the persistence layer
**in-process** (`src/ui/services/urgent_actions.py`), not over HTTP — so
nothing outside the Streamlit process can reach `/api/v2` today.

## Evidence
- `Dockerfile`: single `CMD streamlit run app.py`, `EXPOSE 8501`. `fastapi`
  and `uvicorn[standard]` are already installed in the image
  (`requirements.txt`), so the same image can run the API with a command
  override — no new image or Dockerfile change required.
- `deploy-azure-container-apps.yml`: builds one image to ACR and rolls the
  single Container App via `az containerapp update`.
- The API is fully covered by `TestClient` integration tests
  (`tests/integration/test_api_v2_*.py`), but has never served a real HTTP
  request in any environment.

## Problem
Without a deployed API, every future external consumer is blocked: GIS/BI
tools (the product thesis is to *sit above* ArcGIS/Tableau/Power BI, which
integrate over HTTP), a field-capture app writing `FieldVerification`
records from the #26 campaign, partner parks, or any second internal app.
But deploying it *now* adds a second always-available service, a public
write surface, and recurring cost — with **zero current consumers**.

## Context
- Azure Container Apps supports multiple apps sharing one ACR image, each
  with its own `--command`/`--args` override, ingress port, and
  scale-to-zero profile.
- HTTP ingress on a Container App routes to a single target port, so
  running Streamlit (8501) and uvicorn (8000) inside one app's ingress is
  not cleanly supported.
- Write-auth (5.8) is env-gated: `SNTO_API_KEY` unset = writes open. It is
  currently **intentionally unset** because the only caller is the trusted
  in-process UI. That assumption breaks the moment the API is public.
- Reads are open by design (ADR-011); the audit trail (5.7) records every
  write with an actor.

## Options Considered
- **A. Do not deploy yet (status quo).** In-process consumption covers all
  current functionality. Zero cost, zero new attack surface. Rejected as a
  *permanent* answer (it caps the product thesis) but **accepted as the
  default until a trigger fires** (see Decision).
- **B. Second process inside the existing Container App** (run uvicorn
  alongside Streamlit in one container). Rejected: single-port HTTP ingress
  makes this hacky (needs an internal reverse proxy), couples the two
  services' scaling and failure modes, and saves little.
- **C. Second Container App from the same image** (`snto-api`), command
  override `uvicorn src.api.main:app --host 0.0.0.0 --port 8000`, ingress
  on 8000, scale-to-zero (min replicas 0), same `SNTO_DB_*` secrets, and
  `SNTO_API_KEY` **mandatorily set before first exposure**. **Accepted as
  the target architecture when the trigger fires.** Reuses the existing
  image/CD pipeline (the deploy workflow gains one `az containerapp update`
  line); idle cost ≈ 0 with scale-to-zero.
- **D. Azure Functions / API Management re-platform.** Rejected: off-pattern
  for this codebase (long-lived SQLAlchemy engine, Alembic), new toolchain,
  no benefit at this scale.

## Decision
Adopt **Option A now, Option C when triggered**. The API stays undeployed
until at least one **trigger** materialises:

1. a concrete external consumer (GIS/BI integration, partner park, or any
   client that cannot import the Python package);
2. the #26 field-capture workflow wants to write `FieldVerification`
   records from outside the dashboard; or
3. the v2.0 role-based UI is split into a separate frontend that talks HTTP.

When triggered, deploy as `snto-api` (Option C) with these
**non-negotiable prerequisites**:
- `SNTO_API_KEY` set (writes gated) **before** the first public request;
- narrow DB firewall unchanged (the new app reaches Postgres via the
  existing "Azure services" rule);
- health probe on the existing `/health` endpoint;
- CORS configured only if browser clients appear (none planned);
- the Single-revision secret-update gotcha applies: secret changes need an
  explicit `az containerapp revision restart`.

Sketch (owner-executed, mirrors §4bis style — **no resource is provisioned
by this ADR**):

```bash
az containerapp create -g rg-snto-observatory-app -n snto-api \
    --environment <same-env-as-snto-observatory> \
    --image <same ACR image as snto-observatory> \
    --command uvicorn --args 'src.api.main:app,--host,0.0.0.0,--port,8000' \
    --ingress external --target-port 8000 \
    --min-replicas 0 --max-replicas 1 \
    --secrets snto-db-pass=<...> snto-api-key=<...> \
    --env-vars SNTO_DB_HOST=... SNTO_DB_PORT=5432 SNTO_DB_NAME=snto \
               SNTO_DB_USER=... SNTO_DB_PASS=secretref:snto-db-pass \
               SNTO_API_KEY=secretref:snto-api-key
```

## Consequences
- Positive: the external-integration path is designed, costed (≈ zero idle
  with scale-to-zero; pennies under light use) and prerequisite-checked
  before anyone needs it; no speculative infra runs meanwhile.
- Positive: no code changes are required to execute this ADR — image, CD
  pipeline, auth gate, and health endpoint already exist.
- Negative: until a trigger fires, `/api/v2` remains untested against real
  HTTP traffic (mitigated by the `TestClient` suite, which exercises the
  full ASGI app).
- Negative: when deployed, the deploy workflow must roll **two** apps per
  release or the API image drifts behind the dashboard — the workflow edit
  (one extra `az containerapp update`) must land in the same change.

## Status
**Proposed (2026-07-18) — awaiting owner go/no-go on the trigger.** No
resource is provisioned and no code changes are implied by accepting this
ADR; it pre-approves the architecture so that when a trigger fires, the
deployment is a §4bis-style owner-executed step, not a design discussion.

## Recommendations
Do nothing now. Revisit the moment an external consumer is named — and set
`SNTO_API_KEY` *first* on that day, not after.
