# ADR-011: Concrete Persistent-Backend Architecture for v2.0 (Fase 5)

## Purpose
Concretize ADR-006 ("persistence is required before operational workflows") into an
actual technology choice, resource schema, and phased implementation plan.

## Background
ADR-006 accepted the *need* for persistence but deferred the *design*
("Next Steps: Define the canonical product resources and lifecycle states").
`docs/architecture/future-architecture.md` lists the same requirement as a bullet
point with its own next step: "Convert these requirements into ADRs before major
v2.0 implementation work." Fase 4 (#27) modularized `app.py`; the `app.py`
modularization half of the "Pre-v2" gate (`docs/ux/ui-evolution-v2-spec.md` §11)
is done. The persistence half is not: `.env.example` already declares
`SNTO_DB_HOST/PORT/NAME/USER/PASS` and `requirements.txt` already pins
`sqlalchemy>=2.0` and `psycopg2-binary`, but no model, session, migration, or
connection code exists anywhere in `src/`, and `src/config/settings.py` does not
even read the DB env vars. Persistence was anticipated, never started.

## Evidence
The CTO review (source of ADR-006) named the missing resources explicitly:
assets, observations, alerts, recommendations, interventions, field
verifications, decisions, and audit log. The UX v2.0 spec's central product
object — the **managed asset** with lifecycle
`detected → verified → assigned → funded → resolved → monitored`
(`docs/ux/ui-evolution-v2-spec.md` §3) — cannot exist without exactly this store.

## Problem
"Introduce persistence" is too broad to implement safely in one step. It bundles
technology choice (which DB, which ORM, which migration tool), schema design
(which resources, which lifecycle states, which relationships to the existing
`src/territorial`/`src/assets` Pydantic models), API surface (versioned or not,
which endpoints), and *deployment/cost* decisions (a managed Postgres instance is
a new billable Azure resource; the current Container App is scale-to-zero and
stateless). Conflating these means either stalling on the infra question forever
or an agent unilaterally provisioning cloud resources with real cost — both
wrong.

## Context
This ADR follows the project's own established discipline of *design first, then
implement in small reviewed PRs* (used successfully for the evidence-classes
design in ADR-004's implementation and for the Fase 4 modularization). It
separates what can be decided and built now (schema, code, local/dev
persistence) from what requires an explicit, separate go-ahead (provisioning a
real cloud database, choosing an auth/SSO provider, changing the deployed
Container App's architecture).

## Options Considered
- **Do nothing until v2.0 UI work starts** — rejected: the UX spec's own
  roadmap table names persistence as a v2.0 *prerequisite*, not a parallel
  track; starting UI work first would repeat the "redesigning over the
  monolith" mistake the spec itself warns against (§12), just one layer down.
- **SQLite file under `data/` (already git-ignored), no server** — rejected as
  the permanent answer (no concurrent-write safety, no real multi-user
  story — wrong instrument for an "operational institutional system" per the
  project's positioning), but **accepted as the local/CI persistence target for
  Fase 5's first implementation steps**, exactly the role `USE_MOCK_DATA=true`
  plays for the EO pipeline today: a real, fully-tested code path that swaps to
  the production resource via configuration.
- **PostgreSQL + PostGIS, `sqlalchemy>=2.0` ORM, Alembic migrations** — accepted
  as the target technology. `.env.example` and `requirements.txt` already commit
  to Postgres/SQLAlchemy; PostGIS is the natural fit for the asset geometries
  already modeled as GeoJSON (`src/assets/models.GeoJSONGeometry`); Alembic is
  the SQLAlchemy-native migration tool the project doesn't have yet.
- **Auth: institutional SSO (Azure AD / Entra ID, given the UCM tenant already
  used for OIDC deploy federation) vs. a minimal built-in auth** — **resolved
  2026-07-13 (owner decision): start with minimal built-in auth** (API key or
  session token gating *write* endpoints only; read endpoints stay open, same
  posture as today's stateless API). SSO/Entra ID integration is deferred to
  when real institutional multi-user access is needed — it is an additive swap
  of the auth dependency, not a schema or API-contract change, so deferring it
  costs nothing structurally.

## Decision
Adopt **PostgreSQL + PostGIS** as the target persistence technology,
**SQLAlchemy 2.0** (already a dependency) as the ORM, and **Alembic** (new
dependency) for migrations. Model the ADR-006 resource list as first-class
SQLAlchemy models under a new `src/persistence/` package, built and fully tested
against a **local SQLite file** in dev/CI first — mirroring the project's
existing `USE_MOCK_DATA` pattern — with the production PostgreSQL connection
gated behind `src/config/settings.py` env vars that already exist in
`.env.example` but are not yet read. **No cloud database is provisioned by this
ADR or its Fase 5 implementation PRs** — production Postgres provisioning
remains a separate, explicit, owner-executed step (commands documented in the
Fase 5 plan) whenever the owner is ready to cut over. Write endpoints are
gated by a minimal built-in auth (API key/session token); SSO is deferred.

## Consequences
Positive: unblocks the UX v2.0 spec's central object (managed asset with
lifecycle) and the "Urgent actions" P1 module (`docs/ux/ui-evolution-v2-spec.md`
§6) without any infra/cost commitment up front; every migration and model is
testable in CI exactly as the rest of the suite is today. Negative: a
SQLite-backed dev path and a Postgres-backed prod path must both stay correct
(mitigated by using only SQLAlchemy-portable column types and running the same
test suite against both in CI once the schema stabilizes); the real production
cutover still requires a separate, explicit decision (provisioning, migration
run, connection secrets) before v2.0 can operate on real persisted data.

## Status
**Accepted (2026-07-13) · Implemented (2026-07-18) · Production cutover
executed (2026-07-18).** Concretizes ADR-006. Auth strategy and first UI
consumer resolved (see Decision above and Fase 5 plan). All Fase 5
implementation steps 5.1–5.9 are merged to `main` (#62–#70) against SQLite;
no cloud resource was provisioned by any PR. The owner then executed the
§4bis cutover manually: Azure Postgres Flexible Server `snto-db` (v16,
PostGIS, Sweden Central) is live, migrated via Alembic, and wired into the
Container App via `SNTO_DB_*` secrets. Known gap: the FastAPI `/api/v2` HTTP
surface is not deployed — only the in-process Streamlit consumer uses the
persistence layer today (see Fase 5 plan §4bis for details).
