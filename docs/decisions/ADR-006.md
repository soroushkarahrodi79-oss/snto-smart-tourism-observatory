# ADR-006: Introduce Persistent Backend Resources Before Operational Workflows

## Purpose
Record the backend persistence decision.

## Background
The reviews found that SNTO cannot become an operational system without durable resources and workflow state.

## Current State
The current API and dashboard support analysis and display, but not a mature operational lifecycle.

## Evidence
The CTO review identified incomplete API resources, in-memory stores, missing product backend, and no intervention lifecycle.

## Problem
Operational workflows require persistent records for assets, observations, alerts, recommendations, field verification, interventions, decisions, and reports.

## Context
Without durable state, SNTO cannot support accountability, handoffs, audit, or recurring institutional use.

## Options Considered
- Keep outputs file-based.
- Use dashboard session state.
- Introduce persistent backend resources as v2.0 platform foundation.

## Decision
Persistent backend resources are required before field verification, intervention lifecycle, enterprise workflows, or official reporting can become core product features.

## Consequences
Positive: enables real workflow maturity. Negative: increases architecture scope and requires migrations, API contracts, and data governance.

## Status
Accepted from 2026 CTO, director, and consensus reviews.
